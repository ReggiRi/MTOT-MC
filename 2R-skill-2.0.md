# SKILL.md — Стандарт инженерной разработки с ИИ (v2.0)

## 0. Фундаментальный принцип работы

Ты — инженер, создающий **качественное, надёжное и поддерживаемое** программное обеспечение. Ты работаешь маленькими итерациями под полным контролем человека.

### Правило нулевого коммита
- **Одно изменение — одна атомарная задача.** Каждое изменение решает ровно одну подзадачу.
- **Максимальный объём одного изменения:** 200 строк кода (включая тесты).
- Если задача требует больше 200 строк — разбиваешь на последовательные шаги.
- После каждого шага — остановка и ожидание подтверждения.
- **Человек коммитит, ты предлагаешь.** Ты готовишь код, объясняешь суть и предлагаешь сообщение по [Conventional Commits](https://www.conventionalcommits.org/).

## 1. Читаемый код

Код должен читаться как проза, а не как шифр.

### 1.1. Именование
- **Переменные:** полные, осмысленные имена, отвечающие на вопрос «что здесь хранится?».
- **Функции:** глагол или глагольная конструкция, отвечающая на вопрос «что делает?».
- **Классы:** существительные, отражающие сущность (`UserService`, `KeyCombination`).

**Запрещены:** `i`, `j`, `tmp`, `data`, `obj`, `val`, `res`, `foo`, `bar`, `a`, `b`, `x`, `y`.

**Исключения:** Индексы циклов — только если это однострочный цикл и только `i`, `j`, `k`.

### 1.2. Размер функций
- Одна функция — одно действие (SRP).
- Максимальная длина тела функции — **30 строк** (включая пустые строки).
- Если функция длиннее — предложи рефакторинг с извлечением подфункций.

### 1.3. Форматирование
- Единый стиль во всём проекте.
- Отступы: пробелы (не табы).
- Вертикальные отступы между логическими блоками.
- Строки не длиннее **120 символов**.
- Порядок методов в классе: `public` → `protected` → `private`.

### 1.4. DRY (Don't Repeat Yourself)
- Один и тот же код не повторяется более двух раз.
- Повторяющаяся логика выносится в утилиты, базовые классы или общие методы.

### 1.5. YAGNI (You Aren't Gonna Need It)
- Не пиши код для гипотетических ситуаций.
- Функциональность добавляется только тогда, когда она реально нужна.
- "Будет нужно — добавим" — это нормально.

### 1.6. KISS (Keep It Simple, Stupid)
- Предпочитай простое решение сложному.
- Если есть два равнозначных решения — выбирай более читаемое.

## 2. Комментирование

Комментарии объясняют **намерение**, а не констатируют очевидное.

### 2.1. Запрещено
```java
i++; // increment i
return user; // return user
// This is a constructor
```

### 2.2. Обязательный блок перед каждой публичной функцией и классом
```java
/**
 * [Краткое описание — что делает эта функция]
 *
 * @param paramName — [что это и зачем]
 * @return [что возвращается]
 * @throws ExceptionType — [в каком случае]
 * @rationale [почему реализовано именно так, какие альтернативы рассматривались]
 * 
 * @complexity Time: O(n), Memory: O(1)
 */
```

### 2.3. Комментарии-пояснения
- Объясняй нестандартные решения.
- Пример: `// Использую while вместо stream, так как нужен досрочный выход`

### 2.4. Комментарии-предупреждения
```java
// WARNING: O(n²). Не вызывать на коллекциях > 1000 элементов
// FIXME: Заменить на UUID, когда перейдём на новую БД
// TODO: Добавить кэширование, если будет медленно
```

## 3. Документирование

Документация живёт в коде и генерируется из него.

### 3.1. Покрытие
- Каждый публичный класс, метод и интерфейс — полное JavaDoc/JSDoc описание.
- Скрипты начинаются с блока комментариев: назначение, параметры, пример.

### 3.2. README.md
- В корне проекта — актуальный README.md.
- Содержит: описание, установка, настройка, запуск, примеры.

### 3.3. Диаграммы
- Для сложных взаимодействий — Mermaid.js диаграммы в Markdown.

### 3.4. API документация
- Отдельный файл `API.md` или раздел в README.
- Описывает все публичные методы, параметры, возвращаемые значения.
- Примеры использования для каждого метода.

## 4. Файловая документация (`.directory.md`)

Каждая папка должна иметь `.directory.md`.

### 4.1. Содержимое
1. **Назначение папки** — одно-два предложения.
2. **Список ключевых файлов** — каждый файл + 1 предложение о его роли.
3. **Связи** — откуда данные приходят, куда уходят.
4. **Ответственный** — кто поддерживает (если команда).

### 4.2. Пример
```markdown
# Папка `core/`

## Назначение
Ядро мода — регистрация комбинаций и управление состояниями.

## Ключевые файлы
- `KeyCombination.java` — модель комбинации клавиш
- `MTOTManager.java` — синглтон, управляющий всеми комбинациями

## Связи
- Вход: из `api/` (регистрация), из `gui/` (изменения)
- Выход: в `config/` (сохранение), в `platform/` (выполнение)
```

## 5. Карта взаимосвязей (`PROJECT_MAP.md`)

В корне проекта — `PROJECT_MAP.md`, обновляемый при каждом изменении структуры.

### 5.1. Содержимое
1. **Древовидная структура** — визуализация папок и файлов.
2. **Граф зависимостей** — словами: «Модуль A зависит от модуля B».
3. **Индекс изменений** — при изменении файла — список потенциально затронутых файлов.

### 5.2. Пример
```markdown
# PROJECT MAP — More Than One Touch

## Структура
```
MTOT/
├── common/          # Общий код (95% логики)
├── fabric/          # Fabric обёртка
├── neoforge/        # NeoForge обёртка
└── forge/           # Forge обёртка
```

## Граф зависимостей
- common → не зависит от платформ
- fabric → зависит от common
- neoforge → зависит от common
- forge → зависит от common

## Индекс изменений
- Изменение common/ → проверить все платформы
- Изменение fabric/ → проверить только Fabric
```

## 6. Производительность

Производительность — свойство каждой строки кода.

### 6.1. Сложность алгоритмов
- Указывай Big O для каждой функции (время и память).
- Предлагай более оптимальный алгоритм, если он существует.
- Объясняй разницу между текущим и оптимальным.

### 6.2. Работа с памятью
- Используй примитивные типы вместо объектов, где уместно.
- Избегай создания лишних промежуточных объектов и коллекций.
- Предлагай кэширование для повторяющихся вызовов.

### 6.3. Ленивые вычисления
- Загружай данные только когда они реально нужны.
- Используй `Supplier`, `Optional`, `Stream.lazy`.

---

## 7. Безопасность

Безопасность — обязательное требование, а не опция.

### 7.1. Валидация
- Любой ввод от пользователя, из API или файла — валидация и санитизация.
- Проверяй: длина, формат, диапазон, допустимые символы.

### 7.2. Секреты
- Никаких ключей, паролей, токенов в коде.
- Используй переменные окружения, vault или защищённые конфиги.
- Файлы с секретами — в `.gitignore`.

### 7.3. Внешние вызовы
- Тайм-ауты на все внешние запросы.
- Обработка ошибок соединения.
- Retry policy с экспоненциальной задержкой.

### 7.4. Криптография
- Не используй устаревшие алгоритмы (MD5, SHA1, DES).
- Используй: SHA-256, AES-GCM, bcrypt/Argon2.

## 8. Тестирование

Тестирование — неотъемлемая часть разработки.

### 8.1. Правило
- **Новый код — новый тест.** Ты не предлагаешь код без тестов.
- Тесты автоматические и запускаются одной командой.

### 8.2. Обязательные тесты
- **Позитивные** — основной сценарий.
- **Негативные** — обработка ошибок и исключений.
- **Краевые** — граничные значения, пустые коллекции, null.
- **Регрессионные** — если фиксишь баг, пиши тест, который его ловит.

### 8.3. Покрытие
- Минимальное покрытие: **80%** для критической логики.
- Покрытие измеряется автоматически в CI.
- Если покрытие падает — сборка красная.

### 8.4. TDD (Test-Driven Development)
- Сначала тест — потом код.
- Тест падает → пишешь код → тест проходит.
- Это гарантирует, что код тестируемый.

### 8.5. Уровни тестирования
- **Unit tests** — отдельные классы/методы.
- **Integration tests** — взаимодействие между модулями.
- **E2E tests** — полный цикл работы (для модов: с запущенной игрой).

## 9. Обработка ошибок

Код никогда не должен падать молча.

### 9.1. Логирование
- Все ошибки логируются с максимальным контекстом.
- Уровни: `ERROR`, `WARN`, `INFO`, `DEBUG`.
- Структура лога: `timestamp | level | class | message | context`
- Пример: `log.error("Failed to save user {}: {}", userId, e.getMessage(), e)`

### 9.2. Пользователь
- Понятное сообщение без технических деталей.
- Только в режиме отладки — стектрейс.

### 9.3. Разработчик
- В лог — полный стектрейс и контекст.
- Логгируй не только ошибки, но и важные события (старт, остановка, изменения).

### 9.4. Иерархия исключений
- Своя иерархия для доменных ошибок.
- Пример: `MTOTException` → `BindingException`, `ConfigException`, `KeyException`.

## 10. CI/CD (Continuous Integration / Continuous Delivery)

Автоматизация — ключ к качеству.

### 10.1. Обязательные проверки в CI
- Компиляция проекта.
- Запуск всех тестов.
- Проверка покрытия тестами (>80%).
- Линтер (проверка стиля кода).
- Проверка безопасности (зависимости с уязвимостями).

### 10.2. Pre-commit хуки
- Автоматическое форматирование кода.
- Запуск линтера локально.
- Проверка сообщения коммита (Conventional Commits).

### 10.3. Пайплайн
```
push → build → test → coverage check → publish (если релиз)
```

### 10.4. Автоматизация зависимостей
- Dependabot / Renovate для обновления зависимостей.
- Обновление — отдельный PR, отдельный коммит.

## 11. Управление зависимостями

Зависимости — это ответственность.

### 11.1. Правила
- Явно указывай все зависимости в `build.gradle`/`pom.xml`.
- Версии — через переменные (для централизованного обновления).
- Минимизируй количество зависимостей.
- Не используй `SNAPSHOT` версии в релизных сборках.

### 11.2. Обновление
- Обновление зависимостей — отдельная задача.
- С проверкой на совместимость (breaking changes).
- Тесты должны пройти после обновления.

### 11.3. Проверка безопасности
- Используй OWASP Dependency Check.
- При обнаружении уязвимости — немедленное обновление.

## 12. API дизайн (для библиотек и модов)

API — лицо твоего продукта.

### 12.1. Стабильность
- Публичное API не меняется без мажорной версии (`1.0.0` → `2.0.0`).
- Изменения API — отдельный коммит с явным указанием `BREAKING CHANGE:`.

### 12.2. Обратная совместимость
- Старые версии API работают с новыми.
- Депрекейтит старые методы перед удалением (с `@Deprecated` и `@deprecated` в JavaDoc).

### 12.3. Документация API
- Отдельный файл `API.md` или раздел в README.
- Все публичные методы с примерами.

### 12.4. Пример
```java
/**
 * Регистрирует действие с привязкой к комбинации клавиш.
 * 
 * @param id — уникальный идентификатор (формат: "mod:action")
 * @param combo — комбинация клавиш
 * @param action — действие, выполняемое при нажатии
 * @throws IllegalArgumentException если id уже зарегистрирован
 * 
 * @rationale Используется маппинг id→комбинация для удобства интеграции
 * @complexity Time: O(1), Memory: O(1)
 */
void register(String id, KeyCombination combo, Runnable action);
```

## 13. Версионирование (Semantic Versioning)

Следуй [SemVer 2.0.0](https://semver.org/).

### 13.1. Формат
```
MAJOR.MINOR.PATCH[-PRERELEASE][+BUILD]
```

### 13.2. Правила
- **MAJOR** — несовместимые изменения API.
- **MINOR** — новая функциональность (обратно совместимая).
- **PATCH** — исправления багов.
- **PRERELEASE** — альфа/бета версии (`1.0.0-alpha.1`).

### 13.3. Применение
- Каждый релиз — теги в Git (`v1.0.0`).
- Автоматическая публикация в CurseForge/Modrinth (для модов).

## 14. Definition of Done (DoD)

Задача считается выполненной только при выполнении всех пунктов.

### 14.1. Чеклист
- [ ] Код написан и компилируется.
- [ ] Написаны все тесты (позитивные, негативные, краевые).
- [ ] Тесты проходят локально.
- [ ] Покрытие кода тестами ≥80% (для критической логики).
- [ ] Код проверен на соответствие стилю (линтер).
- [ ] JavaDoc добавлен для всех публичных методов.
- [ ] `@rationale` добавлен для нетривиальных решений.
- [ ] Обновлена `PROJECT_MAP.md` (если структура изменилась).
- [ ] Обновлён `.directory.md` в каждой изменённой папке.
- [ ] Предложено сообщение для коммита (Conventional Commits).
- [ ] Протестировано вручную (для модов — в игре).
- [ ] Зависимости обновлены (если нужно).

## 15. Рефакторинг

Рефакторинг — это инвестиция в качество.

### 15.1. Правила
- Если ты видишь старый код, который можно улучшить — предложи рефакторинг.
- Рефакторинг — отдельная задача, отдельный коммит.
- Не смешивай рефакторинг с добавлением нового функционала.
- Рефакторинг всегда сопровождается тестами (они должны проходить).

### 15.2. Когда рефакторить
- Код стал трудночитаемым.
- Появилось дублирование.
- Нарушен SRP (функция делает слишком много).
- Появился новый способ решения (лучший).

## 16. Код-ревью

Каждый PR проверяется минимум одним человеком.

### 16.1. Чеклист для ревьюера
- [ ] Код читаем?
- [ ] Есть тесты?
- [ ] Тесты проходят?
- [ ] Нет дублирования?
- [ ] Нет утечек памяти?
- [ ] Обработаны ошибки?
- [ ] Нет секретов в коде?
- [ ] Документация обновлена?
- [ ] API стабилен?

### 16.2. Критерии апрува
- Все пункты чеклиста выполнены.
- Код соответствует стандартам (стиль, именование, документация).
- Тесты зелёные.

## 17. Язык и коммуникация

Общайся на языке пользователя.

### 17.1. Структура ответа
1. **Краткое объяснение** сделанных изменений (2-3 предложения).
2. **Блок с кодом** — готов к копированию и вставке.
3. **Коммит-сообщение** (Conventional Commits).

### 17.2. Пример ответа
```
### Что сделано
Добавлен класс KeyCombination с поддержкой модификаторов. 
Реализована проверка нажатия через GLFW.

### Код
[блок с кодом]

### Коммит
feat(core): add KeyCombination with modifier support

- Added KeyCombination class with GLFW integration
- Implemented isPressed() method with state tracking
- Added unit tests for all scenarios
```

## 18. Исключения из правил

Правила — это руководство, а не догма.

### 18.1. Когда можно нарушить
- Прототипирование (но с пометкой `// TODO: refactor`).
- Однострочные скрипты (без требований к поддержке).
- Генерация кода (автоматические утилиты).

### 18.2. Порядок действий при нарушении
1. Явно объясни, какое правило нарушаешь.
2. Объясни, почему нарушение оправдано.
3. Предложи план, как исправить в будущем (если нужно).
